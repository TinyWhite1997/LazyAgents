# Install LazyAgents

> Install path for `lad` (the daemon) + `la` (the TUI), including running `lad` as a service on systemd, launchd, and Windows Task Scheduler. For day-to-day usage after install, see the [user handbook](book/en/src/install.md) and [Quickstart](book/en/src/quickstart.md).

## System requirements

| | Minimum | Notes |
|---|---|---|
| **Linux** | Ubuntu 22.04 LTS or any glibc ≥ 2.31 host (kernel ≥ 5.10) | `x86_64-unknown-linux-gnu` and `aarch64-unknown-linux-gnu`. A static `x86_64-unknown-linux-musl` build is also shipped for Alpine / scratch containers; aarch64 musl is tracked as a v1.x follow-up. |
| **macOS** | macOS 13 (Ventura) or macOS 14 (Sonoma) | `x86_64-apple-darwin` and `aarch64-apple-darwin`. macOS 12 and older are not tested. |
| **Windows** | Windows Server 2022 / Windows 10 build 1809 or later | `x86_64-pc-windows-msvc`. ConPTY is the only supported PTY backend on Windows. |
| **Terminal** | Any ANSI/VT-capable emulator | Kitty, Alacritty, WezTerm, iTerm2, Windows Terminal. |
| **`git`** | 2.20 or newer on `$PATH` | Only required if you use the worktree-review feature. `lad` shells out to `git worktree` / `git apply` / `git diff-tree`. |
| **Backend CLI** | One of `claude`, `codex`, `opencode` already installed and logged in | LazyAgents drives the CLI you authenticate with; it never holds API credentials itself. |

You do **not** need a system SQLite (LazyAgents bundles it), Node.js, Python, or any container runtime.

## Install paths

LazyAgents ships v1 through three install paths. Pick one.

### 1. Tarball / zip (cargo-dist installer scripts)

These are the cargo-dist generated `installer.sh` / `installer.ps1` shipped with every GitHub Release. They download the matching tar.xz (Unix) or zip (Windows) archive, drop `la` and `lad` into your `$PATH`, and verify the [GitHub Artifact Attestation](https://docs.github.com/en/actions/security-guides/using-artifact-attestations-to-establish-provenance-for-builds) (sigstore-backed, keyless).

```sh
# Linux + macOS
curl --proto '=https' --tlsv1.2 -LsSf \
  https://github.com/TinyWhite1997/LazyAgents/releases/latest/download/lazyagents-installer.sh | sh
```

```powershell
# Windows
powershell -ExecutionPolicy ByPass -c "irm https://github.com/TinyWhite1997/LazyAgents/releases/latest/download/lazyagents-installer.ps1 | iex"
```

You can verify the artifact attestation yourself with `gh`:

```sh
gh attestation verify ./lazyagents-x86_64-unknown-linux-gnu.tar.xz \
  --repo TinyWhite1997/LazyAgents
```

If you prefer to download the archive manually and unpack it yourself, the asset list per release lives at <https://github.com/TinyWhite1997/LazyAgents/releases>. Both binaries inside the archive are stripped + LTO-thin (see `Cargo.toml` `[profile.release]`).

### 2. Package managers

| Manager | Status | Command (when available) |
|---|---|---|
| **Homebrew** (`TinyWhite1997/homebrew-tap`) | **Pending publication.** The formula is generated by every release, but the tap repo is not yet public. Use the installer script until the v1.x bump promotes it to brew core. | `brew install TinyWhite1997/homebrew-tap/lazyagents` |
| **Scoop** | **Not currently shipped.** Tracked as a v1.x follow-up; the bucket will land before brew core / scoop main promotion. | `scoop install lazyagents` |
| **brew core / scoop main** | **v1.x submission target.** We will submit formulae to the upstream repositories once the install-time matrix has shipped on real machines for a full release cycle. | `brew install lazyagents` / `scoop install lazyagents` |

Until both ship, the cargo-dist installer is the supported path on every platform — there is no separate `install.sh` for package-manager users.

### 3. From source (any platform)

You need Rust **1.75** or newer (MSRV). CI pins **1.96.0**, but any toolchain ≥ 1.75 will compile.

```sh
git clone https://github.com/TinyWhite1997/LazyAgents.git
cd LazyAgents
cargo install --path crates/la-daemon --locked
cargo install --path crates/la-tui    --locked
```

`la` and `lad` will land in `~/.cargo/bin`.

## Install `lad` as a service

`lad` runs perfectly well in the foreground (`lad start`) or as a daemonized child (`lad daemonize`), but in production you almost certainly want the OS to keep it alive across logouts and reboots. The `lad install --service <mode>` command writes the unit / plist / task XML, and three independent verbs control the lifecycle.

**Critical contract (M4 brief A2 — "三端动词表").** The three install-time verbs are **always independent**. Do **not** chain `install + enable + start` into a single platform-native invocation (the systemd-only `systemctl --user enable --now` shorthand mixes "register" and "start now" into one verb, which makes the resulting docs unportable). Use the dedicated `lad install` flags below, or the per-platform verbs in the same row.

| Verb | What it does | systemd (Linux) | launchd (macOS) | Windows Scheduled Task |
|---|---|---|---|---|
| **install** | Materialise unit file / plist / task XML on disk and register it with the OS. Does NOT start the daemon and does NOT mark it to launch on next login. | `lad install --service systemd` (writes `~/.config/systemd/user/lad.service`; runs `systemctl --user daemon-reload`) | `lad install --service launchd` (writes `~/Library/LaunchAgents/dev.lazyagents.lad.plist`) | `lad install --service windows-task` (`schtasks /Create /XML <lad-task.xml> /TN \LazyAgents\lad /F`) |
| **enable** | Mark the unit so the OS starts it on next login. Does NOT start it now. | `lad install --service systemd --enable` (runs `systemctl --user enable lad.service`) | `lad install --service launchd --enable` (runs `launchctl bootstrap gui/$UID ~/Library/LaunchAgents/dev.lazyagents.lad.plist`) | `lad install --service windows-task --enable` (runs `schtasks /Change /TN \LazyAgents\lad /ENABLE`) |
| **start** | Start the daemon now. Independent of `enable` — you can `--start` without `--enable` (one-shot run that the OS will NOT relaunch on next login). | `lad install --service systemd --start` (runs `systemctl --user start lad.service`) | `lad install --service launchd --start` (runs `launchctl kickstart -k gui/$UID/dev.lazyagents.lad`; auto-bootstraps the plist if `--enable` was not used) | `lad install --service windows-task --start` (runs `schtasks /Run /TN \LazyAgents\lad`) |
| **stop** | Stop the running daemon now. Idempotent — succeeds if already stopped. | `systemctl --user stop lad.service` | `launchctl kill SIGTERM gui/$UID/dev.lazyagents.lad` | `schtasks /End /TN \LazyAgents\lad` |
| **disable** | Mark the unit so the OS does NOT start it on next login. Idempotent. | `systemctl --user disable lad.service` | `launchctl bootout gui/$UID <plist>` | `schtasks /Change /TN \LazyAgents\lad /DISABLE` |
| **uninstall** | Remove the unit/plist/task XML from disk and from the OS registry. Reverses **install**. | `lad uninstall --service systemd` (runs `disable → stop → uninstall` in order) | `lad uninstall --service launchd` (same chain) | `lad uninstall --service windows-task` (same chain) |

**Do** combine the install-time verbs explicitly when you want the all-in-one effect:

```sh
# Linux — install unit, enable for next login, start right now.
lad install --service systemd --enable --start

# macOS — same shape; the daemon is wired into your gui/<uid> domain.
lad install --service launchd --enable --start

# Windows — same shape. Note the user-logon constraint below.
lad install --service windows-task --enable --start
```

**Do not** use the systemd-only `--now` shorthand (`systemctl --user enable --now lad.service`). It silently bundles two of the three verbs above and makes it impossible to describe what the resulting docs do on launchd / Windows. We deliberately do not document it and the `lad install` command does not emit it. The A2 contract is enforced in `crates/la-daemon/src/install/cli.rs::run_install` — each `--enable` / `--start` flag triggers a separate, named platform call.

### Per-platform notes

- **systemd (Linux).** Unit is **user-scope** (`systemctl --user`), not system-scope. `lad` does not run as root; the socket and SQLite live under your user's `$XDG_RUNTIME_DIR` / `$XDG_DATA_HOME`. Logs go to your user journal (`journalctl --user -u lad.service`).
- **launchd (macOS).** Installed under `~/Library/LaunchAgents/dev.lazyagents.lad.plist` and bootstrapped into `gui/$UID`. The plist bakes **absolute paths** (`exec_path`, `config_path`, `home_path`, `user`) — launchd does not expand `$HOME` / `~/` / `$XDG_*` inside `<string>` nodes, so the install command refuses to write a plist that still contains those tokens (A3 belt-and-braces). If you move `lad` after installing, re-run `lad install --service launchd` to refresh the plist.
- **Windows Scheduled Task.** Registered at user logon only (R3 / A8). The daemon runs **only while you are logged in to this Windows account**. Log off → `lad` exits. There is no Windows Service mode in v1; the task is the supported deployment shape. The post-install hint repeats this in both English and Chinese.

## Configuration: the resolved path

`lad` reads its config from a single TOML file. The resolution chain — pinned in `crates/la-config/src/paths.rs::resolve_config_path` — is platform-specific.

**Where `lad` reads from (chain order, first existing wins):**

| # | Linux | macOS | Windows |
|---|---|---|---|
| 1 | `$LAZYAGENTS_CONFIG` (env override) | `$LAZYAGENTS_CONFIG` (env override) | `$LAZYAGENTS_CONFIG` (env override) |
| 2 | `$XDG_CONFIG_HOME/lazyagents/config.toml` | `$XDG_CONFIG_HOME/lazyagents/config.toml` | `%APPDATA%\lazyagents\config.toml` |
| 3 | `$HOME/.config/lazyagents/config.toml` | `$HOME/.config/lazyagents/config.toml` | — |
| 4 | — | `$HOME/Library/Application Support/lazyagents/config.toml` *(read-only compatibility entry)* | — |

**Where `lad` writes a fresh default to** (when the daemon has to materialise one): always **entry 3** on POSIX (`$HOME/.config/lazyagents/config.toml`, regardless of `$XDG_CONFIG_HOME`), and entry 2 on Windows (`%APPDATA%\lazyagents\config.toml`). Dotfile sync stays trivial — XDG-honouring and vanilla users end up writing to the same place.

### macOS configuration fallback chain (M4 brief addendum A4)

On macOS, the fallback chain is purposely longer than the Linux chain. The order matters; the **first existing file wins**:

1. **`$LAZYAGENTS_CONFIG`** — explicit env override. Use this when you keep multiple LazyAgents profiles on the same Mac (e.g. work + personal) or when you want `lad install --service launchd` to bake a specific path into the plist.
2. **`$XDG_CONFIG_HOME/lazyagents/config.toml`** — for users who already standardise on XDG on macOS (dotfile-sync setups, mixed Mac/Linux fleets).
3. **`$HOME/.config/lazyagents/config.toml`** — **the recommended default** on macOS. Matches Linux exactly, keeps your config portable, and is the only POSIX path the daemon will ever *write* a fresh default to.
4. **`$HOME/Library/Application Support/lazyagents/config.toml`** — **read-only compatibility entry**. macOS conventionally puts user-scoped app data under Application Support, so if you copied your config from a tool that follows the macOS convention, `lad` will still find and read it. **It is never a write target.** `lad config show` will surface that you are reading from Application Support; copy the file to entry 3 if you want updates to land somewhere standard.

**Recommended setup on a fresh macOS install:**

```sh
# Recommended default — matches Linux, portable across machines.
mkdir -p "$HOME/.config/lazyagents"
cp templates/config.example.toml "$HOME/.config/lazyagents/config.toml"

# Advanced — keep multiple profiles, point at one via env.
export LAZYAGENTS_CONFIG="$HOME/.config/lazyagents/work.toml"
```

If a path is set with `$LAZYAGENTS_CONFIG` and the file does not exist, `lad` will fail loudly — explicit overrides are never silently demoted to a default lookup. See `lad config check --help` and `lad config path --help`.

## Verify your install

```sh
la --version
lad --version
lad doctor
```

`lad doctor` is the install acceptance gate. It prints a check list and exits with a tier-encoded status:

- **Exit 0** = every critical check passed AND every adapter is `Available`. Each line is rendered with `✓`. This is the "install succeeded, everything green" outcome.
- **Exit 2** = critical checks all green, but at least one optional adapter is missing or unauthenticated. Critical lines render with `✓`, the degraded ones render with `!`. The install is fine; that adapter just won't be usable until you fix it (`claude login`, install `codex`, etc.).
- **Exit 1** = a critical check failed (no daemon socket, wrong permissions, missing `git`, state-dir not writable, version mismatch). Critical line renders with `✗`. The installer must surface the failure and bail; do not proceed until you have a green critical row.

A healthy fresh install — **after** the daemon has been started via `lad install --service <mode> --start`, `lad daemonize`, or letting `la` auto-spawn it on first launch — looks like:

```text
✓ daemon socket reachable: /run/user/1000/lazyagents/lad-1.sock
✓ socket permissions: 0600 owner=uid:1000
✓ daemon version: la 0.1.0 (matches client)
✓ state dir writable: /home/alice/.local/share/lazyagents (free: 765.5 GiB)
✓ git available: git version 2.43.0
✓ adapter claude: ok version=2.1.158
✓ adapter codex: ok version=0.49.0
✓ adapter opencode: ok version=1.2.15
```

Every row green → exit `0`, install is done. Lines for adapters you don't have authenticated will render as `!` (e.g. `! adapter codex: Unauthenticated — see https://developers.openai.com/codex/cli`); that drops the exit to `2` but is not a failure — it just means that backend won't be usable until you log in.

If the daemon is **not** running yet, `lad doctor` will report it as a **critical** failure (exit `1`), not as healthy:

```text
✗ daemon socket reachable: no daemon listening at /run/user/1000/lazyagents/lad-1.sock; start with `lad start` or `lad daemonize`
✗ socket permissions: could not stat /run/user/1000/lazyagents/lad-1.sock: No such file or directory (os error 2); daemon may not have created it yet
✓ daemon version: client la 0.1.0; daemon unreachable, skipping match
✓ state dir writable: /home/alice/.local/share/lazyagents (free: 765.5 GiB)
✓ git available: git version 2.43.0
…
```

Start the daemon (`lad daemonize`, or install it as a service with `lad install --service <mode> --start`) and re-run `lad doctor` — the two `✗` rows should flip to `✓` and the exit code should drop to `0` or `2`.

For ongoing production diagnostics — Prometheus metrics, JSON structured logs, the adapter-drift surface — see [`observability.md`](observability.md). `lad doctor` is the install-time gate; `lad metrics` is the runtime scrape surface.

## Update / uninstall

To upgrade between releases, re-run the installer script for your platform — the script overwrites in place. The full upgrade path (including v1.x → v1.y and the v1 → v2 multi-version coexistence plan) is in [`upgrade.md`](upgrade.md).

### Forks / air-gapped networks: `LAZYAGENTS_UPDATE_MANIFEST_URL`

`la --check-update` queries `https://api.github.com/repos/TinyWhite1997/LazyAgents/releases/latest` by default. If you maintain a self-hosted fork, or your machines cannot reach `api.github.com` (corporate proxy, isolated network), set `LAZYAGENTS_UPDATE_MANIFEST_URL` to point at your own mirror. The endpoint must return a JSON document compatible with the GitHub Releases API — at minimum the fields `tag_name`, `html_url`, and `prerelease`; the optional `name` field is honoured for display.

```sh
# Permanently — drop into your shell profile.
export LAZYAGENTS_UPDATE_MANIFEST_URL=https://git.internal.example.com/api/v4/projects/12/releases/latest

# Or one-shot, against a Gitea / Forgejo mirror that proxies your fork's releases.
LAZYAGENTS_UPDATE_MANIFEST_URL=https://forge.example.com/api/v1/repos/team/LazyAgents/releases/latest \
  la --check-update
```

Network failures are non-fatal — `la --check-update` prints a short stderr note and exits `0` (see `crates/la-tui/src/update_check.rs`), so a wrapper script that bakes the flag in will not start failing the moment your mirror is unreachable.

To remove LazyAgents entirely:

```sh
# Linux / macOS
lad uninstall --service systemd      # or launchd, if installed as a service
rm -f ~/.cargo/bin/la ~/.cargo/bin/lad      # or ~/.local/bin/{la,lad}
rm -rf ~/.local/share/lazyagents             # SQLite + transcripts + worktrees (your data!)
rm -rf ~/.config/lazyagents                  # UI preferences + config.toml
```

```powershell
# Windows
lad uninstall --service windows-task
Remove-Item -Path "$env:USERPROFILE\.cargo\bin\la.exe","$env:USERPROFILE\.cargo\bin\lad.exe"
Remove-Item -Recurse -Force "$env:APPDATA\lazyagents"
Remove-Item -Recurse -Force "$env:LOCALAPPDATA\lazyagents"
```

LazyAgents does **not** write into your backends' data directories. Uninstalling will not touch `~/.claude/projects/`, `~/.codex/sessions/`, or `~/.local/share/opencode/` — those belong to those tools.
