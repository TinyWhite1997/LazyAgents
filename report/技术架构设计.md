## 1. 全景架构

### 1.1 物理拓扑（单机）

```
┌────────────────────────┐                   ┌──────────────────────────────────────────────┐
│  TUI Client (la)       │   length-prefixed │  Daemon (lad)                                │
│  ratatui + crossterm   │ ◀───────────────▶ │  ┌──────────────────────────────────────┐    │
│  attach/detach 任意次  │   JSON-RPC 2.0     │  │ rpc::server                          │    │
│  纯展示 + 输入转发      │   over UDS/Named   │  │ sessions::manager / pty::supervisor  │    │
│                        │   Pipe             │  │ scheduler::cron                      │    │
│  ── 多 client 可同时   │                   │  │ adapters::{claude,codex,opencode,…}  │    │
│     attach 同一 session│                   │  │ storage::sqlite (WAL)                │    │
└─────────┬──────────────┘                   │  │ events::bus (broadcast)              │    │
          │                                  │  │ observ::{tracing, metrics}           │    │
          │                                  │  └──────────────┬───────────────────────┘    │
          │                                  │                 │ spawn / signal / waitpid   │
          │                                  └─────────────────┼────────────────────────────┘
          ▼                                                    ▼
   tty/PTY (本地终端)                          ┌────────────┬──────────────┬───────────────┐
                                               ▼            ▼              ▼
                                          claude code    codex         opencode  …
                                          (每个 session 各自一个 git worktree)
                                          ($XDG_DATA_HOME/lazyagents/wt/<sess>)
```

- **进程数量**：`lad` 守护进程 1 个；`la` 客户端 ≥1 个（可同时多 attach）；每个 agent session 1 个子进程 + 1 个 PTY。
- **网络**：默认**完全无网络监听**。daemon↔client 走 UDS（Unix）/ Named Pipe（Windows）。可选 `--listen 127.0.0.1:<port>` 仅供 IDE 集成，需显式 token。

### 1.2 数据流（一次"打开会话"为例）

```
la 用户按 ⏎  →  la 发送 sessions.attach{session_id} (JSON-RPC)
              ↘ lad 校验 → 订阅该 session 的 stdout/stderr 事件总线
              ↘ 立即回放最近 N KiB 缓冲（"快照"），再追流式增量（"补丁"）
              ↘ 用户输入 → la 发 sessions.write{bytes} → lad 写入 PTY master
              ↘ session 退出 → lad 发布 SessionExited{code} → la 渲染状态
```

关键不变量：**la 永远不直接持有 PTY**。"关闭 TUI 不杀任务"由此保证。

## 2. Crate / 模块划分

单 cargo workspace，二进制 + 库分离便于复用与测试。

```
lazyagents/
├── Cargo.toml                  # [workspace]
├── crates/
│   ├── la-proto/               # ★ 协议契约（serde 结构 + JSON Schema 导出）
│   ├── la-ipc/                 # ★ 传输层：UDS / Named Pipe + length-prefix framing
│   ├── la-pty/                 # ★ 跨平台 PTY 抽象（封装 portable-pty）
│   ├── la-adapter/             # ★ Agent Adapter trait + 内置 adapter 实现
│   │   ├── src/claude.rs
│   │   ├── src/codex.rs
│   │   └── src/opencode.rs
│   ├── la-storage/             # ★ SQLite + sqlx，迁移、仓储、事务边界
│   ├── la-scheduler/           # ★ Cron 引擎、catch-up、并发与花费上限
│   ├── la-core/                # ★ Session/Worktree 管理、事件总线
│   ├── la-observ/              # tracing layer、metrics、健康检查
│   ├── la-daemon/              # bin: `lad`（组装 core + scheduler + ipc + storage）
│   └── la-tui/                 # bin: `la`（ratatui + crossterm；纯 client）
└── xtask/                      # 构建脚本、JSON Schema 生成、跨平台打包
```

### 2.1 依赖原则

- **`la-tui` 不依赖** `la-core` / `la-storage`，只依赖 `la-proto` + `la-ipc`。强制"客户端是纯展示层"。
- **`la-adapter` 不感知 IPC**，只面向 `la-pty` + `la-storage`（写会话元数据）。可单元测试。
- **`la-core` 是组合层**，对外只暴露异步命令接口与事件总线，不直接被 bin 之外的 crate 依赖。
- 共享依赖（`tokio`, `serde`, `tracing`）通过 workspace `[workspace.dependencies]` 锁版本。

### 2.2 运行时

- **Tokio multi-thread runtime**（worker = `min(num_cpus, 8)`）：所有 IO 与 PTY 读循环。
- PTY read 用 `spawn_blocking`（`portable-pty` 的 reader 接口在 Windows ConPTY 下不是真 async），把字节投递到 tokio `mpsc<Bytes>`。
- 调度器与 RPC server 共用 runtime；SQLite 写入串行化在专用 `sqlx::SqlitePool` 上（`max_connections=1` for writes，多个 read pool）。

## 3. ADR-001 — daemon ↔ client 协议选型

> PRD §6.5 开放问题：自定义 socket 协议 vs gRPC vs JSON-RPC？

### 决策

**采用 JSON-RPC 2.0 over length-prefixed framing（4-byte big-endian length + UTF-8 JSON payload），承载在 UDS（Unix）/ Named Pipe（Windows）之上。** 双向：请求/响应 + 服务器主动 notification。版本通过 `initialize` 握手协商。

### 备选与权衡

| 选项 | 优点 | 缺点 | 评分 |
|------|------|------|------|
| **JSON-RPC 2.0 + UDS** | 标准、可读、易调试（`socat`+`jq`）；schema 用 `schemars` 自动导出；多语言客户端零门槛 | JSON 解析比二进制慢；schema 漂移需纪律 | ★★★★★ |
| 自定义二进制（postcard/bincode） | 体积小、解析快 | 难调试；无生态；改协议成本高 | ★★ |
| gRPC | 强类型、streaming 原生 | 拉入 protoc 工具链、tonic 依赖重；UDS 支持有但 Windows Named Pipe 需要桥接；二进制 frame 不利调试 | ★★★ |
| LSP 风格（Content-Length header + JSON） | 现成生态、`tower-lsp` 可参考 | 多了一行 header 解析，无实质收益 | ★★★★ |

JSON-RPC 在**调试性、跨语言、低工具链负担**三者上明显占优；体量上 LazyAgents 单会话峰值流量 < 100 KiB/s（PTY 字节流之外），JSON 解析不是瓶颈。

### 帧格式

```
+--------+----------------------------------+
| u32 BE | UTF-8 JSON-RPC message (≤ 4 MiB) |
+--------+----------------------------------+
```

> 消息大小硬上限 4 MiB（防止恶意客户端 OOM）。PTY 字节流分片：单条 `session.output` notification ≤ 64 KiB，超出自动分片。

### 握手与版本

```jsonc
// → client
{"jsonrpc":"2.0","id":1,"method":"initialize",
 "params":{"client":"la","client_version":"0.4.1","protocol_versions":["1"]}}
// ← daemon
{"jsonrpc":"2.0","id":1,"result":{
  "server":"lad","server_version":"0.4.1","protocol_version":"1",
  "capabilities":{"adapters":["claude","codex","opencode"],"cron":true,"worktree":true}}}
```

**版本策略**：协议主版本通过 `protocol_version` 字段；新增字段为非破坏；删除/重命名需升主版本，daemon 同时支持 N 与 N-1 一个 minor 周期。

### 核心方法集（节选）

> 完整 schema 由 `xtask gen-schema` 从 `la-proto` 导出到 `docs/schema/`。下列只列签名与语义。

| 方法 | 方向 | 参数 / 返回 | 语义 |
|------|------|-------------|------|
| `initialize` | C→D | 见上 | 握手，必须最先调用 |
| `shutdown` | C→D | `{}` | 礼貌断开（daemon 不退出） |
| `sessions.list` | C→D | `{project?,backend?,include_archived?}` → `[SessionSummary]` | 列会话 |
| `sessions.create` | C→D | `{project_dir,backend,args,prompt?,worktree?}` → `SessionDetail` | 新建（含 worktree 创建） |
| `sessions.attach` | C→D | `{session_id, replay_bytes?}` → `AttachAck{snapshot_seq}` | 订阅事件流；返回快照截止序号 |
| `sessions.detach` | C→D | `{session_id}` → `{}` | 取消订阅；不影响会话存活 |
| `sessions.write` | C→D | `{session_id, data_base64}` → `{}` | 写 PTY master |
| `sessions.resize` | C→D | `{session_id, cols, rows}` → `{}` | 改 PTY 窗口 |
| `sessions.signal` | C→D | `{session_id, signal:"INT"\|"TERM"\|"KILL"}` → `{}` | 跨平台映射，详见 §6 |
| `sessions.archive` / `sessions.delete` | C→D | `{session_id}` → `{}` | 软删 / 真删 |
| `sessions.import` | C→D | `{backend, source_path?}` → `[ImportedSession]` | 见 §4.2 发现策略 |
| `crons.list` / `crons.upsert` / `crons.delete` / `crons.run_now` | C→D | 见 §5 | cron 管理 |
| `runs.list` | C→D | `{cron_id?, since?, limit?}` → `[RunRecord]` | 历史回看 |
| `events.subscribe` | C→D | `{topics:[...]}` → 持续 notification | 状态栏全局事件 |
| **服务器 notification** | D→C | | |
| `session.output` | D→C | `{session_id, seq, data_base64}` | PTY 增量；`seq` 单调 |
| `session.state` | D→C | `{session_id, state, exit_code?}` | starting/running/waiting/exited/errored |
| `cron.fired` | D→C | `{cron_id, run_id, fired_at, status}` | |
| `daemon.health` | D→C | `{queue_depth, running, errors_last_5m}` | 状态栏 |

### 关键不变量

- **单一写者**：同一 session 同时只有"一个写者"由 daemon 强制——首个 `attach` + 显式 `acquire_input=true` 取得写权，其他 attach 是只读旁观者，避免多 client 同时输入串流。
- **背压**：客户端订阅默认有 1 MiB 缓冲；溢出时丢弃旧的 `session.output`，并发 `session.gap{from_seq,to_seq}` 让客户端按需 `sessions.replay`。
- **重连**：客户端断线 daemon 保留订阅 30 秒；重连后用上次 `seq` 续传。

### 可推翻条件

> 若 IDE/Web 客户端规模化（多语言客户端 ≥3、QPS 显著上升），重新评估 gRPC + 生成代码的收益。当前 v1 不在路径上。

## 4. ADR-002 — Agent Adapter 抽象与会话发现

### 4.1 Adapter trait

```rust
#[async_trait]
pub trait AgentAdapter: Send + Sync {
    /// 元数据，用于 UI 与能力探测
    fn descriptor(&self) -> AdapterDescriptor;

    /// 探测后端是否安装/可用（执行 `claude --version` 等），不抛错
    async fn probe(&self) -> ProbeResult;

    /// 启动新会话：返回构造好的子进程命令 + 期望环境 + 工作区策略
    fn spawn_spec(&self, req: &SpawnRequest) -> Result<SpawnSpec, AdapterError>;

    /// 发现本机已有会话（读对应后端的会话存储）
    async fn discover(&self, hints: &DiscoverHints) -> Result<Vec<DiscoveredSession>, AdapterError>;

    /// 把"发送 prompt"翻译为写入 PTY 的字节序列（不同后端的多行/结束符差异）
    fn encode_user_input(&self, text: &str) -> Bytes;

    /// 从 PTY 字节流中解析后端的结构化事件（若该后端支持 JSON 输出模式）
    /// 默认返回 Passthrough（纯文本流），无需所有后端都实现
    fn parse_chunk(&self, chunk: &[u8], st: &mut ParserState) -> Vec<AdapterEvent> { … }

    /// 优雅停机序列（不同后端的退出指令）
    fn graceful_stop(&self) -> StopSequence;
}

pub struct SpawnSpec {
    pub program: PathBuf,           // 解析后的绝对路径
    pub args: Vec<OsString>,
    pub env: Vec<(OsString, OsString)>,
    pub cwd: PathBuf,               // = worktree 路径
    pub pty: PtyHints,              // 初始 cols/rows、ICANON 是否关闭
    pub stdin_mode: StdinMode,      // Pty / NullSink (cron 无人值守)
}

pub struct DiscoveredSession {
    pub external_id: String,        // 后端自己的 session id
    pub project_hint: Option<PathBuf>,
    pub created_at: DateTime<Utc>,
    pub title_hint: Option<String>,
    pub source: DiscoverySource,    // FileSystem{path} | Sqlite{path} | CliQuery
}
```

> Adapter **不感知 IPC、不感知 SQLite**：返回 spec、消费字节、产出事件。这样能用 `cargo test` 单元跑。

### 4.2 各后端发现 / 导入策略（v1）

| Backend | 发现位置 / 命令 | 字段映射 | 边界处理 |
|---------|----------------|----------|----------|
| **Claude Code** | `~/.claude/projects/<hash>/<session>.jsonl`（JSONL 每行一条消息） | 首条 message.timestamp → created_at；project 路径反查 hash | 后端未登录 → `probe` 返回 `Unauthenticated`，UI 灰态 |
| **Codex CLI** | `~/.codex/sessions/*.json` 或 `codex sessions list --json`（优先 CLI 查询，文件结构作为 fallback） | meta.cwd → project_hint | CLI 不存在 → `Missing`；版本差异以最小公共字段为准 |
| **OpenCode** | `$XDG_DATA_HOME/opencode/sessions/*.json` | project 字段直读 | 同上 |

**导入策略选择（开放问题答复）：**

- **v1 默认：双轨并行**——既"接管"本工具新建的会话，也"只读发现"既有会话；既有会话可一键"导入"成为受管 session（迁移元数据到 SQLite，原文件保留只读引用）。
- **不强制吞并**既有数据，避免触发用户对其它 CLI 的数据所有权焦虑。
- **会话续接**：被发现的会话**不能续接 PTY**（原进程已退；其本地存储是"对话历史"，非活进程）；点击"resume"等同于让 adapter 用"恢复模式"启新进程（如 `claude --resume <id>`），由 adapter 自行处理。

### 4.3 错误分类

```rust
pub enum AdapterError {
    NotInstalled { hint: String },          // probe 阶段
    Unauthenticated { docs_url: String },   // probe / spawn
    SpawnFailed(std::io::Error),
    UnsupportedOption { name: String },
    ProtocolDrift { detail: String },       // 后端 JSON 输出格式变了
    Transient(String),                      // 可重试
}
```

`ProtocolDrift` 必须 emit `error_event::adapter_drift` 指标 + 高优先级日志，提示用户升级。

### 4.4 加新 adapter 的成本

只需 (1) 实现 `AgentAdapter`，(2) 在 `la-adapter::registry` 注册，(3) 在 `xtask schema` 自动暴露到 UI 选项。不动 daemon 主干。这是把"多后端"作为底线（PRD §6 决策3）的工程兑现。

## 5. ADR-003 — Cron 调度器

### 5.1 调度模型

- **Cron 表达式**: 支持 5 字段（`m h dom mon dow`）与可选 6 字段（前置秒）。解析与下一次触发使用 `cron` crate（成熟、纯 Rust）。
- **时区**: 每条 cron 自带 `tz` 字段（IANA 名），默认 daemon 启动时的本地时区；保存 IANA 名而非 offset，DST 安全。
- **持久化**: 所有 cron 与 run 记录落 SQLite（schema 见 §7）。daemon 内存只缓存"下次触发时间堆"。

### 5.2 调度循环

```
loop {
    let next = heap.peek();                       // (next_fire_at, cron_id)
    select! {
        _ = sleep_until(next.fire_at) => fire(next.cron_id),
        cmd = ctl_rx.recv() => apply(cmd),        // upsert/delete/run_now 立即重排
        _ = shutdown_rx.recv() => break,
    }
}
```

> 使用 `tokio::time::sleep_until` 而非 `tokio::time::interval`，避免漂移。
> 时钟跳变监测：每 60 s 比对 `Instant::now()` 与 `SystemTime::now()` 漂移；偏差 > 30 s 时重算所有 next_fire。

### 5.3 错过触发（catch-up）补偿

**策略：可配置三选一，默认 `coalesce`。**

| 模式 | 行为 | 适用场景 |
|------|------|----------|
| `skip` | 错过的全部丢弃；下一次按计划 | 纯巡检任务 |
| **`coalesce`（默认）** | 错过 N 次只触发 1 次（最近一次），标注 `coalesced_count=N` | 大多数 idempotent 任务 |
| `replay` | 错过 N 次依次触发（节流：每次间隔 ≥ `min_interval_s`） | 历史补跑、报表 |

启动时计算每条 cron 的 `missed_runs = past_fires_between(last_fired_at, now)`，按上述模式入队。`max_catchup` 硬上限 = 100，超出强制 `skip` 并写告警事件。

### 5.4 并发与花费上限

每条 cron 独立配额：

| 字段 | 默认 | 含义 |
|------|------|------|
| `max_concurrent_runs` | 1 | 同一 cron 上一次未结束时是否允许重叠（默认禁止） |
| `max_runs_per_day` | 24 | 24h 滚动窗口硬上限 |
| `max_runtime_s` | 1800 | 单次运行超时强杀 |
| `cost_budget_usd_per_day` | null | 若 adapter 能上报 token/费用，则到上限熔断 |
| `failure_backoff` | `expo(1m, 2x, cap=1h)` | 连续失败时延后下一次触发 |
| `pause_on_consecutive_failures` | 5 | 连续失败到阈值自动停用，UI 高亮提示 |

全局上限（防一次配置错误打爆主机）：

- `daemon.scheduler.global_max_concurrent_runs = 8`
- `daemon.scheduler.cpu_load_throttle = 4.0`（1 分钟 loadavg 超过则推迟触发）

### 5.5 运行生命周期

```
Pending → Spawning → Running → (Completed | Failed | TimedOut | Cancelled | BudgetExceeded)
                              ↘ Outputs 持久化到 runs.transcript_path
```

- 每次 run 在 `runs` 表创建一行（见 §7）。开始时间、结束时间、退出码、coalesced_count、cost 估算全部记录。
- **失败记录**保留 stdout/stderr 尾部 64 KiB + 完整 transcript 文件路径；UI 上能 `r` 重新触发该 run（用相同参数）。
- 触发的 session 与正常人工 session **隔离同等**：独立 worktree、独立子进程；只是 `origin = "cron:<cron_id>"`，且 stdin 模式默认 `NullSink`（无人值守不能发等待输入的 prompt）。
- 完成后 worktree **不自动删除**：留给用户评审 diff；UI 提供"接受"（合并到 base 分支）/"丢弃"（删 worktree）按钮。

### 5.6 防"偷偷烧钱"

- 状态栏永远显示 **下一个 cron 触发时间** 与 **今日累计触发次数 / 估算成本**。
- 创建 cron 时默认 `enabled = false`，需用户显式 `space` 启用——第一次启用前弹出确认：列出预算与下一次触发时间。
- 提供 `crons.dry_run` 让用户预演前 5 次触发时间。

### 5.7 可推翻条件

> 若用户出现"我要每秒跑一次"或"我要数千条 cron"的需求，迁移到 `tokio-cron-scheduler` 或独立调度 binary；当前默认能撑 ~1000 条 cron / 日 ~10k 次触发，对个人工具足够。

## 6. ADR-004 — 跨平台 PTY 与子进程生命周期

### 6.1 抽象层

**采用 `portable-pty` 作为底层**，在 `la-pty` crate 内提供一层薄包装：屏蔽 ConPTY/unix-pty 差异，只暴露 `tokio` 友好的 `PtyChild { reader: mpsc::Receiver<Bytes>, writer: PtyWriter, resize, signal, wait }`。

> 备选：直接调 `windows::Win32::System::Console` + `nix::pty`。自己实现学习曲线陡、维护负担大；`portable-pty` 已经被 wezterm 在产品中验证多年。**不重新发明**。

### 6.2 写循环

```rust
let (out_tx, out_rx) = mpsc::channel(1024);   // PTY → consumer
let reader = pair.master.try_clone_reader()?;
tokio::task::spawn_blocking(move || {
    let mut buf = [0u8; 8192];
    loop {
        match reader.read(&mut buf) {
            Ok(0) => break,                    // EOF
            Ok(n) => {
                if out_tx.blocking_send(Bytes::copy_from_slice(&buf[..n])).is_err() {
                    break;                     // consumer dropped → 退出读循环
                }
            }
            Err(e) if e.kind() == ErrorKind::Interrupted => continue,
            Err(_) => break,
        }
    }
});
```

- **`blocking_send` 的背压**：consumer 跟不上 → 读循环阻塞 → PTY 缓冲填满 → 子进程 write 被 OS 阻塞。这正是我们要的（不要无界缓冲导致 OOM）。
- session manager 侧把 `out_rx` 落 ring buffer（默认 2 MiB / session）+ 广播给 attached client；ring buffer 满则覆盖最旧字节并打 `session.gap` 标记。

### 6.3 进程生命周期

| 阶段 | 跨平台动作 |
|------|------------|
| **spawn** | `CommandBuilder` 设 cwd、env、program；`pair.slave.spawn_command(cmd)` |
| **resize** | `pair.master.resize(PtySize { rows, cols, .. })`（ConPTY 与 unix 一致） |
| **signal** | Unix: `nix::sys::signal::kill(pid, SIG…)`；Windows: 见下表 |
| **wait** | 专用 `spawn_blocking` 调 `child.wait()`，把 `ExitStatus` 投递事件总线 |
| **orphan 清理** | daemon 启动时扫描 `sessions` 表 `state in (Running, Starting)`、`pid` 在系统中不存在的，标 `Exited{unknown}` |

#### Windows 信号映射

| 语义 | Unix | Windows ConPTY |
|------|------|----------------|
| 用户 Ctrl-C | SIGINT | `GenerateConsoleCtrlEvent(CTRL_C_EVENT, pid_group)` |
| 礼貌终止 | SIGTERM | `GenerateConsoleCtrlEvent(CTRL_BREAK_EVENT, pid_group)` |
| 强杀 | SIGKILL | `TerminateProcess(handle, 1)` |

> 子进程必须放入独立 process group（Unix：`setsid` / Windows：`CREATE_NEW_PROCESS_GROUP`），否则 Ctrl-C 会广播到 daemon 自己。`portable-pty` 在新版本默认行为不一致，需在 `la-pty` 显式设置并测试。

### 6.4 优雅停机序列

`graceful_stop`（adapter 自定义，默认序列）：

1. 调 adapter 的"退出指令"（如 claude 的 `/exit`），等待 3 s；
2. 发 SIGTERM / CTRL_BREAK，等待 2 s；
3. 强杀；
4. 清理 PTY（drop master/slave）；
5. 写 `sessions.state = Exited{code, reason}`。

整个序列在 daemon 关闭时对所有 session 并发执行，硬超时 10 s。

### 6.5 已知陷阱

- **Windows ConPTY 的 ANSI 序列差异**：ConPTY 注入额外的 cursor query 与 OSC，需要在 client 渲染层用 `vte` parser 容忍。已在 wezterm/zellij 等项目验证。
- **Unix 终端属性继承**：daemon 进程没有 controlling tty，spawn 时必须用 `setsid`，否则部分 agent CLI 会拒绝启动。
- **环境变量泄漏**：daemon 进程的 env 不应原封不动传给子进程。`SpawnSpec.env` 是白名单 + adapter 显式覆盖；`PATH` 经过净化。

## 7. SQLite Schema

### 7.1 设计约束

- **文件位置**：`$XDG_DATA_HOME/lazyagents/lad.sqlite`（Unix）/ `%LOCALAPPDATA%\LazyAgents\lad.sqlite`（Windows）。权限 0600 / 仅当前用户。
- **WAL 模式**：`PRAGMA journal_mode=WAL; PRAGMA synchronous=NORMAL; PRAGMA foreign_keys=ON;`
- **迁移**：`sqlx migrate`，迁移文件 `migrations/00xx__*.sql` 顺序号严格递增，CI 校验"前向兼容（旧 daemon 读新 schema 不崩）"通过加列+默认值。
- **写入串行化**：写 pool max=1，避免 SQLite writer 锁竞争；读 pool max=4。
- **UUID**：v7（带时间戳，便于按时间排序），存为 TEXT。

### 7.2 表结构（DDL 节选）

```sql
-- 后端 adapter 元数据快照（启动时根据注册表 upsert）
CREATE TABLE backends (
  id              TEXT PRIMARY KEY,             -- 'claude' | 'codex' | 'opencode' ...
  display_name    TEXT NOT NULL,
  version         TEXT,                         -- probe 时填
  available       INTEGER NOT NULL DEFAULT 0,
  last_probed_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 项目（按目录归一化路径分组）
CREATE TABLE projects (
  id              TEXT PRIMARY KEY,             -- uuid v7
  root_path       TEXT NOT NULL UNIQUE,         -- canonicalized absolute path
  display_name    TEXT NOT NULL,
  vcs             TEXT,                         -- 'git' | null
  created_at      TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_projects_root ON projects(root_path);

-- 会话：人工 + cron 触发 都落这里
CREATE TABLE sessions (
  id              TEXT PRIMARY KEY,
  project_id      TEXT NOT NULL REFERENCES projects(id) ON DELETE RESTRICT,
  backend_id      TEXT NOT NULL REFERENCES backends(id),
  external_id     TEXT,                         -- 后端自己的 session id（导入时存）
  title           TEXT,                         -- 用户可编辑
  state           TEXT NOT NULL,                -- 'starting'|'running'|'waiting'|'exited'|'errored'|'archived'
  exit_code       INTEGER,
  pid             INTEGER,
  worktree_path   TEXT,                         -- null = 共享主工作区（不推荐）
  worktree_branch TEXT,
  base_branch     TEXT,
  spawn_args      TEXT NOT NULL,                -- json: 启动参数快照
  origin          TEXT NOT NULL DEFAULT 'user', -- 'user' | 'cron:<cron_id>' | 'import'
  created_at      TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at      TEXT NOT NULL DEFAULT (datetime('now')),
  archived_at     TEXT
);
CREATE INDEX idx_sessions_project ON sessions(project_id) WHERE archived_at IS NULL;
CREATE INDEX idx_sessions_state ON sessions(state);
CREATE INDEX idx_sessions_origin ON sessions(origin);

-- transcript：增量追加，避免单行巨大
CREATE TABLE session_chunks (
  session_id      TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  seq             INTEGER NOT NULL,
  ts              TEXT NOT NULL DEFAULT (datetime('now')),
  kind            TEXT NOT NULL,                -- 'stdout' | 'stderr' | 'input' | 'event'
  data            BLOB NOT NULL,
  PRIMARY KEY(session_id, seq)
) WITHOUT ROWID;
-- 注：当 transcript 累计 > 8 MiB 时滚动到外部文件 sessions/<id>.log，DB 只留指针 + 索引

-- cron 定义
CREATE TABLE crons (
  id                          TEXT PRIMARY KEY,
  name                        TEXT NOT NULL,
  enabled                     INTEGER NOT NULL DEFAULT 0,
  project_id                  TEXT NOT NULL REFERENCES projects(id),
  backend_id                  TEXT NOT NULL REFERENCES backends(id),
  spawn_args                  TEXT NOT NULL,    -- json
  prompt                      TEXT NOT NULL,
  cron_expr                   TEXT NOT NULL,
  tz                          TEXT NOT NULL,    -- IANA
  catchup_mode                TEXT NOT NULL DEFAULT 'coalesce',  -- skip|coalesce|replay
  max_concurrent_runs         INTEGER NOT NULL DEFAULT 1,
  max_runs_per_day            INTEGER NOT NULL DEFAULT 24,
  max_runtime_s               INTEGER NOT NULL DEFAULT 1800,
  cost_budget_usd_per_day     REAL,
  failure_backoff             TEXT NOT NULL DEFAULT 'expo(1m,2,1h)',
  pause_on_consecutive_failures INTEGER NOT NULL DEFAULT 5,
  consecutive_failures        INTEGER NOT NULL DEFAULT 0,
  last_fired_at               TEXT,
  next_fire_at                TEXT,
  created_at                  TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at                  TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_crons_next ON crons(next_fire_at) WHERE enabled = 1;

-- 每次触发的运行记录
CREATE TABLE runs (
  id              TEXT PRIMARY KEY,
  cron_id         TEXT REFERENCES crons(id) ON DELETE SET NULL,
  session_id      TEXT REFERENCES sessions(id) ON DELETE SET NULL,
  scheduled_at    TEXT NOT NULL,                -- cron 表达式应触发时刻
  started_at      TEXT,
  finished_at     TEXT,
  status          TEXT NOT NULL,                -- pending|spawning|running|completed|failed|timed_out|cancelled|budget_exceeded
  exit_code       INTEGER,
  coalesced_count INTEGER NOT NULL DEFAULT 1,   -- 合并的错过次数
  cost_usd_est    REAL,                         -- 若 adapter 上报
  error_kind      TEXT,                         -- AdapterError 分类
  error_detail    TEXT,
  tail_log        BLOB                          -- stderr 尾部 ≤ 64 KiB
);
CREATE INDEX idx_runs_cron_time ON runs(cron_id, scheduled_at DESC);
CREATE INDEX idx_runs_status ON runs(status);

-- 用户偏好 / 设置
CREATE TABLE settings (
  key             TEXT PRIMARY KEY,
  value           TEXT NOT NULL
);

-- 协议/schema 版本登记，便于 daemon 升级时校验
CREATE TABLE schema_meta (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
```

### 7.3 数据生命周期

- `runs`：默认保留 90 天，超期归档到 `runs/archive/<yyyymm>.jsonl.zst` 后删除（任务每日 03:17 触发）。
- `session_chunks`：会话归档时迁出到外部文件；删除时级联清理。
- 全库支持 `lad backup --output <path>`，生成一致快照（用 SQLite Online Backup API）。

### 7.4 迁移示例（M3 增 cost 字段）

```sql
-- migrations/0007__cron_cost_budget.sql
ALTER TABLE crons ADD COLUMN cost_budget_usd_per_day REAL;
ALTER TABLE runs  ADD COLUMN cost_usd_est REAL;
```

迁移幂等性由 `sqlx::migrate!` 保证；上线前在 CI 验证"老 schema 升新 schema 后 daemon 启动正常"。

## 8. ADR-005 — git worktree 隔离与生命周期

### 8.1 命名与位置

- worktree 根：`$XDG_DATA_HOME/lazyagents/worktrees/<project_slug>/<session_short_id>`
- 分支名：`la/<session_short_id>` 由 `base_branch` 派生。

### 8.2 创建流程

```
git -C <project_root> worktree add --detach <wt_path> <base_sha>
git -C <wt_path> checkout -b la/<sid>
```

- 失败回退：删除 wt 路径、报错。
- `base_branch` 默认 = `origin/HEAD` 解析后的远端默认分支；无网时用本地 `git symbolic-ref HEAD`。

### 8.3 复用与缓存（缓解 PRD §6.4 worktree 依赖痛点）

- v1 不做"依赖共享卷"（容易引发版本冲突）；改为 **post-create hook 机制**：用户可配置 `.lazyagents/hooks/post-worktree-create.sh`，在 wt 创建后自动跑（如 `pnpm install --frozen-lockfile`、`uv sync`）。
- 缓存目录共享走包管理器自己的全局缓存（pnpm store、cargo registry、uv cache），通过环境变量传递。
- Later：探索 reflink / overlayfs 复制 node_modules（M4+，非 v1 范围）。

### 8.4 清理策略

- 会话 `archived` 后保留 wt 7 天，过期由后台任务 `git worktree remove`。
- 用户强制清理：`la sessions purge --older-than 30d`。
- `git worktree prune` 启动时跑一次，回收外部删除留下的孤儿。

## 9. 错误处理 & 可观测性

### 9.1 错误类型骨架

```rust
#[derive(Debug, thiserror::Error)]
pub enum DaemonError {
    #[error("ipc: {0}")]      Ipc(#[from] IpcError),
    #[error("adapter: {0}")]  Adapter(#[from] AdapterError),
    #[error("storage: {0}")]  Storage(#[from] sqlx::Error),
    #[error("pty: {0}")]      Pty(#[from] PtyError),
    #[error("scheduler: {0}")]Scheduler(#[from] SchedulerError),
    #[error("config: {0}")]   Config(String),
    #[error("invariant violated: {0}")] Invariant(&'static str),
}
```

- `Invariant` 用于"理论上不可能"的情况，触发即 dump traces 并 panic（仅 daemon 内部）。
- 所有 IPC 暴露错误经 `to_rpc_error(code, message, data)` 映射为 JSON-RPC `error` 对象，**绝不把内部 panic 跨进程透传**。
- 错误码段：`-32000..-32099` 保留给协议；`-33000..` 为业务错误（adapter/scheduler/storage 各分段）。

### 9.2 重试策略

| 场景 | 策略 |
|------|------|
| Adapter `Transient` | 指数退避 1s→8s，最多 3 次 |
| SQLite `SQLITE_BUSY` | 抖动重试 50ms→500ms，最多 8 次 |
| PTY EAGAIN | 当作正常背压，不重试 |
| Cron 触发失败 | 进 `failure_backoff` 流程，不立刻重试 |

### 9.3 可观测性栈

- **结构化日志**：`tracing` + `tracing-subscriber` 输出 JSON 行；默认 `INFO`，可 `lad --log-level debug`。
- **trace 关联**：每次 RPC 调用、每次 cron 触发分配 `trace_id`（128bit），写入所有日志与 `runs.tail_log` 头部。
- **指标**：`metrics` crate，导出到 `lad metrics` 端点（仅 UDS）；关键指标：
  - `lad_rpc_requests_total{method,result}`
  - `lad_session_active`
  - `lad_session_output_bytes_total{backend}`
  - `lad_cron_runs_total{status}`
  - `lad_pty_spawn_duration_seconds`
  - `lad_storage_write_latency_seconds`
- **健康检查**：`daemon.health` notification 每 5s 推送；`la doctor` 子命令做一次性体检（依赖 / 权限 / 版本）。
- **崩溃报告**：`color-backtrace` + `panic_hook` 写 `crashes/<ts>.json`，包含最近 100 条 tracing 事件（in-memory ring）；用户可选择上报。

## 10. 安全模型

### 10.1 威胁模型

- **同主机其他用户读取 IPC**：本地账户隔离前提下不应能连接。
- **本机恶意进程冒充 client**：限制为"已知执行体" hash 不现实（升级后变化），转而靠**进程同 UID + UDS 文件权限**。
- **跨网络访问**：默认无网络监听；开启 TCP 模式必须显式 token，否则拒绝启动。
- **agent 写到任意路径**：worktree 隔离 + 强烈不建议 `--allow-fs-everywhere`，UI 显式警示。
- **secrets 泄漏**：transcript 与 cron 历史不写入用户键入的明显敏感字段（密码框），但**不**做内容扫描；文档明确告知"日志可能含 prompt 全文"。

### 10.2 IPC 权限

| 平台 | 措施 |
|------|------|
| Unix | socket 文件 0600，父目录 0700；启动时 `umask(0077)`；连接后 `SO_PEERCRED` 校验 UID = daemon UID（不等则拒绝） |
| Windows | Named Pipe 仅当前用户 SID（`PIPE_REJECT_REMOTE_CLIENTS`）；ACL 拒绝其他用户 |

### 10.3 配置与凭据

- LazyAgents 自身**不存储**任何 agent 后端的凭据：依赖各 CLI 自身的鉴权（用户在系统里跑过 `claude auth login` 等）。
- 我们写入的设置文件不超过 1024 字节，仅 UI 偏好、cron 元数据；任何带秘密的字段（如可选 webhook URL with token）单独标 `SecretString` 类型，日志中以 `***` 渲染。

### 10.4 cron 安全

- 创建 cron 默认 `enabled=false`，首次启用强制弹确认（防止恶意 `multica issue` style 注入构造危险任务）。
- 拒绝在调度 spawn 时使用 shell wrapping；spawn 始终用 `execve` 路径调用，不走 `/bin/sh -c`。

### 10.5 升级与签名

- 发布二进制经 cosign 签名；`la --check-update` 可选拉取 manifest，**默认不自动升级**（避免 cron 任务运行中切换 daemon 版本）。

## 11. 配置与部署

### 11.1 配置文件

`$XDG_CONFIG_HOME/lazyagents/config.toml`（TOML，因为 daemon 重启不频繁，可读性优先）：

```toml
[daemon]
socket_path = "default"          # default = OS 标准位置
log_level   = "info"
listen_tcp  = ""                 # 留空 = 关闭

[scheduler]
global_max_concurrent_runs = 8
cpu_load_throttle          = 4.0

[worktree]
post_create_hook = ".lazyagents/hooks/post-worktree-create.sh"
prune_after_days = 7

[adapters.claude]
command = "claude"               # 解析 PATH；可绝对路径
extra_args = []

[adapters.codex]
command = "codex"
prefer_json_mode = true

[ui]
theme = "auto"
key_hints = "rich"
```

### 11.2 daemon 启动模式

| 模式 | 描述 |
|------|------|
| 手动 | `lad start` 前台运行 |
| 后台 | `lad daemonize`（fork + setsid） |
| systemd | 安装包附带 `lazyagents.service` user unit |
| launchd | macOS user agent plist |
| Windows | 后台计划任务（用户登录时启动） |

`la` 启动时若连不上 daemon，自动尝试 `lad daemonize`（除非用户禁用）。

### 11.3 多版本共存

socket 路径包含主版本号（`lad-1.sock`），允许同主机存在 v1 与 v2 daemon 互不冲突。

## 12. 测试策略

- **单元**：`la-adapter` / `la-scheduler` / `la-storage` 走标准 `cargo test`，对 PTY 与时间用 `mockall` + `tokio::time::pause`。
- **协议契约**：`la-proto` 用 `schemars` 导出 JSON Schema → CI 用 `jsonschema` crate 跑 round-trip golden tests，破坏即红。
- **集成**：`integration/` 起一个真实 `lad` 子进程 + 模拟 client，验证完整 RPC 流；模拟 adapter `mock-cli`（一个用 Rust 写的假 claude，可注入慢响应/崩溃）。
- **跨平台 CI**：GitHub Actions matrix `ubuntu-latest / macos-latest / windows-latest`，必须三端绿才能合 main（呼应 PRD 风险表）。
- **PTY 烟雾**：每平台必须能跑 `echo hi` + `cat` 双向通讯 + resize + Ctrl-C 三个最小用例。
- **调度回归**：用 `tokio::time::pause` 跑"模拟 7 天 cron 时间线"，验证 catch-up 模式与 backoff。
- **混沌**：可选 `chaos` 测试套件，随机 kill 子进程、注入 SQLite I/O 错误、断 IPC 连接，验证 daemon 不崩。

## 15. ADR 总览（推翻条件速查）

| ADR | 决策 | 主要取舍 | 推翻条件 |
|-----|------|----------|----------|
| 001 | JSON-RPC 2.0 + UDS/Named Pipe | 调试性 vs 体积 | 跨语言客户端激增 / QPS 显著 |
| 002 | Adapter trait + 双轨发现 | 多后端接入成本低 vs 单一后端深度不足 | 某后端用户体量碾压式占优，可特化 |
| 003 | 自研轻量 cron + coalesce 默认 | 控制力 vs 复杂度 | 用户出现 1000+ cron 规模 |
| 004 | portable-pty | 复用成熟实现 | portable-pty 在 ConPTY 出现严重 bug 且不可绕过 |
| 005 | git worktree + hook | 上手低 vs 依赖重装痛 | 用户研究表明 hook 不足以缓解，需上容器层 |

## 16. 仍未决（v1 期间需进一步研究，但不阻塞 M0/M1）

- **多 client 协作模式**：v1 默认"单写多读"；将来若做"两人同观一个 session 共编 prompt"，需引入冲突合并。
- **Adapter 上报成本/Token 的统一字段**：各家口径不一，§5.4 的 `cost_usd_est` 在 M3 之前先记 token 估算，M4 起再尝试统一货币单位。
- **远端 daemon**：是否允许 SSH 通道把 `la` attach 到远端主机的 `lad`。倾向 v1 不做，由 SSH 转发 UDS（`ssh -L` for unix domain socket，OpenSSH 8.7+）作为非官方支持。
