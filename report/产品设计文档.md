# 产品设计文档：多 Coding Agent 管理 TUI

**状态**: Draft v1.0 ｜ **作者**: 产品 ｜ **最后更新**: 2026-05-31

---

## 0. 一句话定位

一个**终端原生、键盘驱动（vim 操作逻辑）、前后端分离**的多 coding agent 管理工具：把分散在 Claude Code / Codex / OpenCode 等不同 CLI agent 里的会话，按项目目录统一收拢到一个 TUI 管理；并通过常驻 daemon 支持**定时（cron）自动运行 agent**——这是当前同类产品普遍缺失的能力。

---

## 1. 问题陈述（Why now）

开发者正从"一次开一个 AI 编程会话"转向"**同时跑多个 agent**"。三个具体痛点：

1. **会话散落、无统一入口。** Claude Code、Codex、OpenCode 各自独立，会话散在不同终端 / tmux window，跨项目切换靠人脑记忆。
2. **没有调度能力。** 现有工具几乎全是"前台、临时"的——关掉会话就停。没人能做到"每天凌晨让 agent 自动跑测试修复 / 巡检依赖升级"。
3. **优质 UX 都在 GUI，终端用户被亏待。** 体验最好的几款（Conductor、Sculptor、vibe-kanban）全是桌面/Web。唯一有体量的 TUI（claude-squad）缺项目分组、缺调度、强依赖 tmux 且 Windows 易碎。

**竞品调研（2026-05）：**

| 工具 | 形态 | 多后端 | Stars | 关键缺口 |
|------|------|--------|-------|----------|
| claude-squad | TUI(tmux) | 是 | ~7.7k | 无项目分组/cron/daemon，Windows 易碎 |
| vibe-kanban | Web 看板 | 是 | ~26.7k | 非终端原生 |
| Conductor | Mac 原生 | Claude+Codex | — | 仅 Mac、无调度 |
| Crystal/Nimbalyst | Electron | Claude+Codex | ~3.1k | 桌面应用、无调度 |
| Sculptor | 桌面+容器 | Claude+Codex | — | 需 Docker、上手重、无调度 |
| Tmux-Orchestrator | 脚本 | Claude | ~2 | 唯一尝试调度，但是脆弱脚本 |

**结论：** "**键盘驱动 TUI + 项目目录分组 + 一等公民 cron/daemon**" 是真实存在、目前无人占据的空白。再等 6 个月 claude-squad 很可能补上项目分组，窗口会收窄——现在是切入点。

---

## 2. 目标与成功指标（North Star + 支撑指标）

**North Star**：周活跃用户中**由本工具发起的 agent 会话数 / 用户 / 周**（衡量它是否成为真正的日常入口，而非装一次就忘）。

| 目标 | 指标 | 测量窗口 |
|------|------|----------|
| 成为日常入口 | 周人均工具内发起会话数 ≥ 15 | 上线后 60 天 |
| 调度被真正使用 | 创建 ≥1 个 cron 的用户占比 ≥ 30% | 90 天 |
| 跨后端被使用 | 使用 ≥2 种 agent 后端的用户占比 ≥ 40% | 90 天 |
| 留存 | 安装后 W4 留存 ≥ 35% | 首批 cohort |
| 跨平台可用 | Linux/macOS/Windows 三端崩溃率均 < 1% | 持续 |

---

## 3. 非目标（v1 明确不做）

- **不做 agent 本身。** 我们是编排/管理层，不替换 Claude Code/Codex 的推理。后端 agent 由用户自行安装鉴权。
- **不做 Web/桌面 GUI。** v1 只做 TUI（终端原生是我们的身份）。允许鼠标，但定位是键盘优先。
- **不做团队/云协作。** v1 是本地优先单机工具。多人/云编排是 Later。
- **不做容器隔离。** v1 用 git worktree 做会话隔离；Docker 容器隔离作为后续可选层。
- **不内置代码评审合并的复杂工作流**之外的能力（如内置 CI、内置编辑器）；diff 评审够用即可，深度编辑跳转到用户自己的 IDE。

## 4. 用户画像与核心故事

**主画像：终端原住民开发者。** 重度 vim/neovim 用户，日常活在 tmux/zellij + lazygit 里，已在用至少一个 CLI coding agent，讨厌离开键盘、讨厌打开浏览器。

**次画像：平台/运维工程师。** 想让 agent 定时跑巡检、依赖升级、日志分析等无人值守任务。

**核心用户故事（含验收标准）：**

**Story 1 — 统一查看跨后端会话**
> 作为终端开发者，我希望在一个界面里看到所有项目下、所有 agent 后端的会话，这样我不用记它们散在哪。
- [ ] 给定本机存在 Claude Code 与 Codex 的历史会话，启动 TUI 后左栏按项目目录分组列出全部会话，并标注其后端类型。
- [ ] j/k 与方向键可在会话间导航；回车进入会话，右侧主区加载对话。
- [ ] 边界：某后端未安装/未鉴权时，该分组显示灰态提示而非崩溃。

**Story 2 — 管理会话生命周期**
> 作为用户，我希望删除或归档不再需要的会话，保持列表干净。
- [ ] 选中会话按 `d` 触发删除（二次确认）；按 `a` 归档（移入 Archived 分组，可恢复）。
- [ ] 删除/归档为 daemon 侧操作，TUI 断开重连后状态一致。

**Story 3 — 定时自动运行 agent**
> 作为运维，我希望配置"每天 2:00 让 Codex 在项目 X 跑依赖巡检并提 PR"，无需我在场。
- [ ] 在 Crons tab 创建任务：名称、agent 选择、启动参数、cron 表达式、prompt 内容。
- [ ] daemon 在 TUI 关闭时仍按时触发；运行结果产生一个可回看的会话/记录。
- [ ] 边界：触发时后端不可用 → 记录失败原因，不静默丢失。

**Story 4 — 随时连接/断开**
> 作为用户，我希望随时 `tool attach` 连上 daemon 查看进度，关掉 TUI 后台任务继续跑。
- [ ] 关闭 TUI 不杀死正在运行的 agent 会话与 cron。
- [ ] 重新 attach 后能看到运行期间的全部增量输出。

## 5. 方案概览与界面设计

整体借鉴 **OpenAI Codex app（桌面版，非 CLU）** 的"agent 指挥中心"三栏心智，落到 TUI 形态；交互逻辑用 **vim**；底部按键提示借鉴 **lazygit / zellij / which-key.nvim**。

### 5.1 总体布局

```
┌─ Tabs ───────────────────────────────────────────────────────┐
│  [ Sessions ]   Crons                                         │
├──────────────┬───────────────────────────────────────────────┤
│ SIDEBAR      │ MAIN AREA                                      │
│              │                                                │
│ ▾ ~/proj-a   │  (Sessions: 选中会话的对话/日志 + diff 评审)   │
│   ● claude   │  (Crons:    选中 cron 的编辑表单)              │
│   ○ codex    │                                                │
│ ▾ ~/proj-b   │                                                │
│   ● opencode │                                                │
│ ▸ Archived   │                                                │
├──────────────┴───────────────────────────────────────────────┤
│ STATUS BAR: daemon●  3 running · 1 cron due 02:00 · branch…   │
├───────────────────────────────────────────────────────────────┤
│ KEY HINTS:  j/k 导航 · ⏎ 进入 · d 删除 · a 归档 · n 新建 · ? 全部 │
└───────────────────────────────────────────────────────────────┘
```

### 5.2 顶部 Tab 区
- 两个一级 Tab：**Sessions**、**Crons**。`Tab`/`Shift+Tab` 或数字键 `1`/`2` 切换（lazygit 式直达）。
- 预留扩展位（未来：Automations 结果队列、Logs）。

### 5.3 Sessions tab
**左侧边栏：**
- 会话**按 project（目录）分组**，组内列出来自不同后端的 session，行内用图标/色块标注后端（claude/codex/opencode…）与运行状态（● 运行中 / ○ 空闲 / ⏸ 待输入 / ✗ 出错）。
- 导航：`j/k` 或方向键上下；`h/l` 折叠/展开分组；`g/G` 跳首尾。
- 操作：`⏎` 进入会话（右侧主区启动对话）；`d` 删除（确认）；`a` 归档；`n` 在当前项目新建会话（弹出后端选择 + 启动参数）。
- 末尾固定 **Archived** 分组，可展开恢复。

**右侧主区（进入会话后）：**
- 上：与 agent 的**对话/日志流**（流式输出，可滚动 `Ctrl+u/d`）。
- 下/可切换：**git diff 评审面板**（借鉴 Codex review pane）——查看本次改动、`s` 暂存 / `x` 撤销某块、`c` 提交。深度编辑用 `o` 跳转外部 IDE。
- 底部 composer：输入 prompt 发给 agent。

### 5.4 Crons tab
- 左侧边栏：用户创建的 cron 任务列表（`j/k` 导航，`n` 新建，`d` 删除，`space` 启用/停用）。
- 右侧主区：选中 cron 的**编辑表单**，字段：
  - 名称
  - agent 选择（claude/codex/opencode…）
  - agent 启动参数
  - cron 时间（表达式 + 人类可读预览"下次：明日 02:00"）
  - prompt 内容（多行）
- 保存即下发到 daemon；`r` 可手动"立即运行一次"用于测试。

### 5.5 状态栏（Status Bar）
一开始想不到放什么，建议承载**全局健康与待办信号**（这恰是它的价值）：
- daemon 连接状态（●在线/○离线）
- 当前运行中的会话数、即将触发的下一个 cron 及时间
- 当前会话的 git 分支 / 改动数
- 全局错误徽标（有失败任务时高亮，引导去看）

### 5.6 底部按键提示（核心体验）
借鉴 lazygit/zellij/which-key，原则：
1. **上下文驱动**：只显示"当前焦点面板 / 选中项 / 当前模式"下可用的操作。
2. **按重要性排序**，不按字母序（which-key 的 sort 机制证明这是刻意设计轴）。
3. **渐进披露**：常驻一行显示高频操作；按 `?` 弹出当前上下文的**完整按键浮层**。
4. **即时更新**：焦点/模式/选中变化时立刻刷新——它同时是教学工具，用户无需背快捷键。
5. **可选模态**（vim/zellij 式）：如进入"会话操作模式"后单键直达，缩小全局键空间、避免冲突。

### 5.7 鼠标支持
允许鼠标：点击切换 Tab、选中会话、点击 diff 块、滚轮滚动。但所有操作都有键盘等价路径，键盘始终是一等公民。

## 6. 技术架构考量（前后端分离）

### 6.1 架构总览

```
┌──────────────┐   IPC (本地 socket / gRPC)   ┌────────────────────────────┐
│   TUI Client │ ◀───────────────────────────▶ │   Daemon (常驻)            │
│ (可多次 attach│                              │  • Session Manager         │
│  /detach)    │                              │  • Cron Scheduler          │
└──────────────┘                              │  • PTY/进程管理            │
                                              │  • Agent Adapters          │
                                              │  • 状态持久化(SQLite)      │
                                              └───────────┬────────────────┘
                                                          │ 启动/接管子进程
                              ┌──────────────┬────────────┴────────────┐
                              ▼              ▼                         ▼
                         claude code      codex                   opencode …
                         (各在 git worktree 隔离的工作区)
```

### 6.2 关键决策

- **决策1：daemon 直接管 PTY，而非 shell 到 tmux。**
  claude-squad 最高频的 bug 都来自 tmux 耦合（Windows 易碎、复用 tmux server 时新会话不继承自定义环境变量）。我们让 daemon 自己持有伪终端（PTY），跨平台更稳——这是相对头号竞品的具体质量优势。取舍：自己实现 PTY 管理（Windows ConPTY / *nix pty）成本更高，但换来跨平台与健壮。

- **决策2：git worktree 做会话隔离，v1 不上容器。**
  全行业已收敛到"每个 agent 一个隔离工作区 + diff 评审后合并"。worktree 是最低门槛方案。取舍：worktree 复用本地环境、可能需每会话重装依赖——这是行业 #1 抱怨。缓解：自动化 worktree 创建/清理，探索依赖缓存共享；容器隔离留作 Later 可选层。

- **决策3：Agent Adapter 抽象层。**
  统一接口（启动、发送 prompt、流式读取、状态查询、停止），每个后端一个 adapter。多后端是**底线而非卖点**。v1 首发 Claude Code + Codex + OpenCode。

- **决策4：Rust 实现。**
  跨平台单二进制分发、PTY/并发/性能友好、TUI 生态成熟（ratatui）。daemon 与 client 同语言共享协议定义。取舍：开发速度慢于 Go，但分发与跨平台收益大；竞品 claude-squad 用 Go(Bubble Tea)，我们以稳定性与调度差异化而非语言。

- **决策5：状态持久化用 SQLite（daemon 侧）。**
  会话元数据、cron 定义、运行历史落 SQLite，保证 TUI 断开重连一致、daemon 重启可恢复。

### 6.3 跨平台

| 平台 | TUI | Daemon | PTY | 备注 |
|------|-----|--------|-----|------|
| Linux | ✓ | systemd/手动 | pty | 主力 |
| macOS | ✓ | launchd/手动 | pty | 主力 |
| Windows | ✓ | 服务/手动 | ConPTY | 重点验证，避开 tmux 依赖是关键优势 |

### 6.4 已知风险

| 风险 | 可能性 | 影响 | 缓解 |
|------|--------|------|------|
| Windows PTY/进程管理复杂 | 高 | 高 | 第1周做 ConPTY spike 验证；CI 纳入三端 |
| 各 agent CLI 接口不稳定/无结构化输出 | 中 | 高 | adapter 层隔离；优先用各家 headless/JSON 模式；做版本兼容矩阵 |
| worktree 依赖重装拖慢体验 | 高 | 中 | 自动化 + 依赖缓存探索；文档化最佳实践 |
| daemon 安全（本地 socket 被滥用） | 中 | 中 | socket 权限收敛到当前用户；不默认开网络端口 |
| cron 误触发造成 token 烧钱 | 中 | 中 | 状态栏显示下次触发；dry-run；并发上限与花费可见性 |

### 6.5 开发前必须澄清的开放问题
- [ ] 各 agent 后端是否提供稳定的非交互/JSON 输出模式？（决定 adapter 复杂度）—— Owner: 工程 —— 第1周
- [ ] daemon↔client 协议选型：自定义 socket 协议 vs gRPC vs JSON-RPC？—— Owner: 工程 —— 第1周
- [ ] 既有会话如何"发现"导入（读取各家本地会话存储）vs 仅管理本工具新建的会话？—— Owner: 产品+工程 —— 第2周

## 7. 里程碑与发布计划

| 阶段 | 范围 | 成功门槛 |
|------|------|----------|
| **M0 Spike（第1-2周）** | Windows ConPTY 验证；1个 agent adapter 跑通；daemon↔client 协议原型 | PTY 在三端可起停一个 claude 会话；协议能流式回传输出 |
| **M1 核心 TUI（第3-6周）** | Sessions tab：按项目分组、j/k 导航、进入会话对话、删除/归档；底部上下文按键提示 | 单后端下完成 Story 1/2；键位手感对标 lazygit |
| **M2 多后端 + 隔离（第7-9周）** | 接入 Codex/OpenCode；git worktree 隔离 + diff 评审面板 | Story 1 跨 ≥2 后端；diff 评审可暂存/提交 |
| **M3 Daemon + Cron（第10-13周）** | 常驻 daemon、attach/detach、Crons tab、调度执行与历史回看 | 完成 Story 3/4；关闭 TUI 后 cron 仍按时触发 |
| **M4 打磨发布（第14-16周）** | 跨平台分发、主题/紧凑模式、文档、状态栏完善 | 三端崩溃率<1%；onboarding<5分钟 |

**Alpha**：M3 末，团队 + 10 名终端重度用户。
**Beta**：M4 中，公开招募 100 名，错误率<5%、核心流 CSAT≥4/5。
**GA**：M4 末。

## 8. 我们明确不做 + 理由（建立信任、减少重复请求）

| 请求 | 来源 | 暂不做的理由 | 重启条件 |
|------|------|--------------|----------|
| Web/桌面 GUI | 潜在用户 | 终端原生是我们的差异化身份；GUI 赛道已拥挤 | 终端用户验证后、有明确 GUI 需求规模 |
| 团队/云协作 | 设想 | v1 本地优先单机；云编排是另一个产品 | 单机留存达标后 |
| 容器隔离 | 竞品对标 | worktree 足够覆盖 v1；容器抬高上手门槛 | worktree 依赖痛点被证明阻塞采用 |
| 内置编辑器 | 设想 | 用户已有 vim/IDE，重复造轮子 | 不计划 |
| 替换/路由模型后端 | 对标 claude-code-router | 那是另一层（模型路由），非会话管理 | 不计划 |

## 9. 竞争策略小结

- **头条差异化 = cron + 常驻 daemon**：几乎无竞品提供一等公民的定时 agent 运行 + 永远在线的后端。以此为主打。
- **多后端是底线**：首发 Claude Code + Codex + OpenCode。
- **解决继承来的 worktree 痛点**：自动化 + 缓存，至少让它"不可见"。
- **不脆弱依赖 tmux**：daemon 自管 PTY，作为相对 claude-squad 的稳定性硬优势。
- **守住身份**：本地优先 + 终端原生 + 可脚本化，对抗头部玩家向云/团队漂移。
